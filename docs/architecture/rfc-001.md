# 1. Введение

## 1.1 Цель документа

Документ описывает архитектуру первой версии проекта **NodePilot** и фиксирует основные инженерные решения, принятые до начала разработки.

RFC является основным документом проекта и используется для принятия дальнейших архитектурных решений.

---

# 2. Описание проекта

**NodePilot** — open-source система удаленного управления Linux-хостами, ориентированная на разработчиков.

Проект предоставляет единый веб-интерфейс для управления домашними серверами, VPS и другими Linux-устройствами.

В отличие от большинства аналогичных решений, NodePilot строится по принципу **Agent First** — агент является центральным компонентом системы и самостоятельно взаимодействует с операционной системой.

Backend выполняет роль координатора и точки доступа, а Frontend — пользовательского интерфейса.

---

# 3. Проблема

При работе с домашним сервером разработчик регулярно выполняет повторяющиеся действия:

* подключается по SSH;
* проверяет загрузку CPU;
* проверяет использование памяти;
* смотрит состояние сервисов;
* просматривает логи;
* запускает Docker-контейнеры;
* выполняет повторяющиеся Bash-команды.

При наличии нескольких серверов управление становится неудобным из-за отсутствия единой панели управления.

NodePilot должен решить эту проблему.

---

# 4. Цели проекта

NodePilot должен позволять:

* централизованно управлять Linux-хостами;
* получать системные метрики в режиме реального времени;
* выполнять удалённые команды;
* управлять сервисами;
* просматривать журналы;
* работать без постоянного использования SSH;
* легко расширяться новой функциональностью.

---

# 5. Основные принципы

## Agent First

Агент является главным компонентом системы.

Вся платформозависимая логика находится внутри агента.

Backend не взаимодействует напрямую с Linux.

---

## Single Responsibility

Каждый компонент отвечает только за свою область ответственности.

Agent — взаимодействие с ОС.

Backend — маршрутизация сообщений, хранение данных, авторизация.

Frontend — отображение информации.

---

## Persistent Connection

Соединение всегда инициируется агентом.

Backend никогда самостоятельно не подключается к агентам.

Это позволяет:

* работать через NAT;
* избежать открытия входящих портов;
* автоматически переподключаться после потери соединения.

---

## Extensibility

Любая новая возможность должна добавляться без изменения ядра системы.

# 8. Компоненты системы

## Agent

Агент является самостоятельным приложением, постоянно работающим на Linux-хосте.

Он отвечает за:

* сбор системных метрик;
* выполнение команд;
* работу с файлами;
* чтение журналов;
* управление сервисами;
* управление процессами;
* взаимодействие с Linux API;
* выполнение запросов Backend.

Агент является единственным компонентом системы, который знает особенности операционной системы.

---

## Backend

Backend представляет собой центральный координатор системы.

Он отвечает за:

* регистрацию агентов;
* авторизацию пользователей;
* маршрутизацию сообщений;
* хранение состояния;
* REST API;
* WebSocket API.

Backend не должен содержать платформозависимый код.

---

## Frontend

Frontend предоставляет пользователю единый интерфейс управления системой.

---

# 9. MVP

Первая версия должна содержать:

* регистрацию агента;
* подключение к Backend;
* мониторинг CPU;
* мониторинг RAM;
* мониторинг дисков;
* uptime;
* выполнение одной команды;
* просмотр результата;
* просмотр логов;
* управление сервисами.

---

# 10. Протокол

Передача сообщений осуществляется по TCP.

Формат сообщений — JSON.

Каждое сообщение содержит поле

```json
{
    "type": "..."
}
```

## HELLO

Регистрация агента.

## WELCOME

Подтверждение регистрации.

## METRICS

Передача системных метрик.

## COMMAND_REQUEST

Запрос на выполнение операции.

## COMMAND_RESULT

Ответ агента.

## HEARTBEAT

Поддержание соединения.

---

# 11. Принцип взаимодействия

NodePilot использует **агент-центричную модель взаимодействия**.

Backend никогда не обращается напрямую к операционной системе.

Все операции проходят через агент.

```
Browser

↓

Backend

↓

Agent

↓

Linux
```

Даже такие операции, как:

* чтение файлов;
* управление Docker;
* перезапуск сервисов;
* просмотр логов;
* запуск процессов;

выполняются исключительно агентом.

---

# 12. Внутренняя архитектура агента

Агент рассматривается не как один исполняемый файл, а как набор независимых сервисов.

```
Agent
│
├── NetworkManager
├── MetricsCollector
├── CommandExecutor
├── ServiceManager
├── FileManager
├── LogManager
├── PluginManager
├── ConfigManager
└── Scheduler
```

Каждый модуль отвечает только за одну задачу.

Такое разделение позволит независимо развивать функциональность и тестировать отдельные части агента.

---

# 13. Ограничения MVP

Поддерживается только:

* Linux;
* x86_64;
* один Backend;
* один агент на одном хосте.

В первой версии отсутствуют:

* TLS;
* VPN;
* Docker;
* плагины;
* GPU-мониторинг;
* кластеризация.

---
