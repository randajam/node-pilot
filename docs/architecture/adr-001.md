# ADR-001 — Transport Abstraction

> **Статус:** Accepted
>
> **Дата:** 14.07.2026
>
> **Автор:** randajam

---

# Контекст

NodePilot строится по принципу **Agent First**.

Основной задачей агента является выполнение операций над операционной системой Linux.

При этом агент не должен зависеть от конкретного способа передачи данных.

На раннем этапе разработки требуется возможность проверять работу агента без реализации сетевого стека. Позже консольный вывод должен быть заменен настоящим TCP-соединением без изменения логики агента.

Возникает вопрос:

> Должен ли Agent самостоятельно работать с сокетами, или взаимодействие с сетью необходимо вынести в отдельный слой?

---

# Решение

В проект вводится абстракция транспортного слоя.

```text
Agent
    │
    ▼
ITransport
    ▲
    │
ConsoleTransport
TCPTransport
```

Agent взаимодействует исключительно с интерфейсом `ITransport`.

```cpp
class ITransport {
public:
    virtual ~ITransport() = default;
    virtual void send(const std::string& message) = 0;
};
```

Конкретная реализация транспорта определяется на этапе создания приложения.

В MVP используется `ConsoleTransport`, который выводит сообщения в консоль.

В дальнейшем он будет заменен на `TCPTransport`, реализующий передачу данных по сети.

---

# Причины принятия решения

## Независимость Agent от сети

Agent не знает:

- что такое TCP;
- что такое сокет;
- что такое порт;
- что такое IP-адрес.

Он умеет только передавать сообщения.

---

## Возможность тестирования

Во время разработки можно использовать различные реализации транспорта.

Например:

```
ConsoleTransport
```

печатает сообщения в терминал.

```
MockTransport
```

сохраняет сообщения в память и используется в unit-тестах.

```
TCPTransport
```

отправляет данные на Backend.

Agent при этом остается неизменным.

---

## Следование принципу Dependency Inversion

Высокоуровневый модуль (Agent) не зависит от низкоуровневого (TCP).

Оба зависят от абстракции (`ITransport`).

Это уменьшает связанность компонентов системы.

---

## Простота расширения

Добавление нового транспорта не требует изменения существующего кода агента.

Например:

```
WebSocketTransport
```

или

```
UnixSocketTransport
```

могут быть реализованы как новые классы, наследующие `ITransport`.

---

# Последствия

## Положительные

- Agent становится независимым от способа передачи данных.
- Появляется возможность тестирования без сети.
- Снижается связанность компонентов.
- Облегчается дальнейшее развитие проекта.
- Поддерживается принцип Open/Closed.

---

## Отрицательные

- Появляется дополнительный уровень абстракции.
- Усложняется структура проекта по сравнению с прямым использованием сокетов.

Для NodePilot данная сложность считается оправданной.

---

# Рассмотренные альтернативы

## Вариант 1

Agent самостоятельно открывает сокет и отправляет данные.

```
Agent
    │
socket()
connect()
send()
```

### Недостатки

- сильная связанность;
- невозможность использовать другой транспорт;
- усложнение тестирования;
- смешение бизнес-логики и сетевого взаимодействия.

Решение отклонено.

---

## Вариант 2

Передавать в Agent готовый объект TCPTransport.

```cpp
Agent(TCPTransport&)
```

### Недостатки

Agent начинает зависеть от конкретной реализации транспорта.

Любое изменение транспортного слоя потребует изменения Agent.

Решение отклонено.
